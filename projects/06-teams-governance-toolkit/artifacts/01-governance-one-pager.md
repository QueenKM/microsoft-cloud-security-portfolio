# Teams Governance One-Pager

## Context

`Astera University` needs Microsoft Teams collaboration that is easy for departments to use but still controlled enough to manage guests, naming, lifecycle, and stale team cleanup.

## Baseline Decisions

| Question | Default Answer |
| --- | --- |
| Who can own a team? | At least two named internal owners |
| Can users create teams? | Yes, through controlled self-service with request metadata |
| How are teams named? | `Department-WorkspaceType-ShortPurpose` |
| When are guests allowed? | Only with a sponsor and a clear external collaboration need |
| When are shared channels preferred? | When only one external workstream must be shared |
| How are stale teams handled? | Review every 180 days, archive before delete |
| Who owns retention? | Central compliance and governance teams, not individual team owners |

## Owner Checklist

- keep two owners assigned
- review guests quarterly
- archive the team when the work ends
- do not use ad hoc naming or permanent exceptions

## Interview Soundbite

This toolkit turns Teams governance into a repeatable operating model. It does not just say "lock things down"; it defines when to use guest access, when to use shared channels, how to keep owners accountable, and how to clean up stale workspaces before they become an access risk.
